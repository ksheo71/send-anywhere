# 업로드 UX 개선 — 병렬 업로드 + 파일별 진행률 + 전체 취소

작성일: 2026-07-06
상태: 승인됨 (구현 계획 작성 대기)

## 1. 목적과 범위

현재 `web/src/SendPage.tsx`는 파일을 **순차**로 업로드하고 **전체 진행률 바 하나**만 보여준다.
이를 개선한다:

1. **병렬 업로드** — 동시 실행 상한(3)을 둔 풀로 여러 파일을 동시에 올려 속도 개선.
2. **파일별 진행률/상태** — 각 파일 행에 개별 진행률 바 + 상태(대기/업로드중/완료/에러/취소).
3. **전체 취소** — 업로드 중 모두 중단하고 파일 선택 상태로 복귀.

**범위 제약 — 프론트엔드만.** `web/src/api.ts`와 백엔드 `src/**`는 변경하지 않는다
(백엔드 `npm test` 40/40 유지). 검증된 업로드 흐름은 그대로 둔다:
- `createTransfer(files)` → 각 파일 tus 업로드(`metadata.fileId` = 그 파일의 createTransfer id,
  서버 namingFunction이 의존) → 전 파일 완료 후 `finalizeTransfer`.
- 로컬 QR 생성(`qrcode`), 결과 카드.

바꾸는 것은 업로드 오케스트레이션(순차→병렬 풀)과 진행률/취소 UI뿐이다.

## 2. 동작 설계

**업로드 태스크 상태(파일별)**: `queued | uploading | done | error | canceled` + 개별 progress(0~100)
+ 해당 tus.Upload 참조.

**동시 실행 풀**: 상한 `CONCURRENCY = 3`(상수). 최대 3개 파일을 동시에 업로드하고, 하나가 끝나면
다음 대기 파일을 시작한다. (50MB 청크가 과도하게 동시 전송되는 것을 막아 브라우저/네트워크
과부하 방지.)

**전체 진행률** = 모든 파일의 업로드된 바이트 합 / 총 바이트. 기존 전체 진행률 바를 유지하고,
파일별 진행률 바를 추가한다.

**완료**: 모든 파일이 `done`이면 `finalizeTransfer(transferId)` → 결과 카드(기존과 동일).

**에러**: 한 파일이 tus 재시도(`retryDelays`)까지 실패하면 그 파일을 `error`로 표시한다. 진행 중인
다른 업로드는 마저 끝내되, 에러 파일이 있으면 `finalize`하지 않는다(서버 finalize는 전 파일 완료를
요구). "다시 시도"(에러 파일만 tus로 재개 — tus는 오프셋부터 이어받음) 또는 "전체 취소"를 노출.

**전체 취소**: 진행 중인 모든 tus.Upload를 `abort()`하고, 파일 선택 상태로 복귀한다(선택한 파일은
유지, 진행률/상태 리셋). 서버에 반쯤 올라간 전송은 버려지고 24시간 스위퍼가 정리한다. 이후 다시
"보내기"를 누르면 **새 `createTransfer`**로 새 전송을 시작한다(새 fileId들).

## 3. 구조

테스트 가능한 단위로 분리한다.

- **신규 `web/src/lib/uploadPool.ts`** — 순수 동시성 풀 유틸.
  `runPool<T>(items: T[], concurrency: number, run: (item: T, index: number) => Promise<void>): Promise<void>`.
  최대 `concurrency`개를 동시에 실행하고, 하나 끝나면 다음을 시작하며, 모두 끝나면 resolve한다.
  `run`이 throw하면 그 항목은 실패로 간주하되 나머지 항목의 실행은 계속한다(개별 실패가 풀 전체를
  멈추지 않음). 순수 로직이라 vitest로 검증한다.
- **신규 `web/src/components/FileRow.tsx`** — 파일 한 줄 표시. props: 파일명·용량·상태·progress·제거
  콜백(업로드 중 비활성). 상태 아이콘은 lucide(`Loader2` 회전/`CheckCircle2`/`AlertCircle`).
- **`web/src/SendPage.tsx` 재작성** — 위 풀과 FileRow를 사용해 병렬 업로드·파일별 진행률·전체
  취소·에러 재시도를 오케스트레이션. tus.Upload 참조 관리. createTransfer/metadata.fileId/finalize/
  로컬 QR 로직은 유지.

## 4. UI 흐름

파일 드롭/선택 → 파일 목록(각 행에 상태 아이콘 + 개별 진행률 바) → "보내기" →
동시 3개 병렬 업로드(파일별 바 진행 + 전체 진행률 바 + "전체 취소" 버튼) → 모두 완료 시 결과 카드
(6자리 코드·QR·링크). 에러 시 해당 파일 빨간 상태 + "다시 시도"/"전체 취소". 반응형·라이트/다크
유지.

## 5. 검증

- **단위 테스트** `web/src/lib/__tests__/uploadPool.test.ts` (vitest):
  - 동시 실행 수가 `concurrency`를 넘지 않는다.
  - 모든 항목이 실행되고 완료된다.
  - 한 항목이 throw해도 나머지가 모두 실행된다.
- `npm run build:web` 그린, `npm test`는 백엔드 40/40 + 새 uploadPool 테스트 통과.
- **브라우저 확인**: 여러 파일 병렬 업로드 시 파일별 바가 동시에 진행, 전체 취소가 즉시 중단하고
  파일 선택 상태로 복귀, 에러 파일 재시도, 완료 → 코드/QR/링크. 라이트/다크.
- 배포는 확인 후 `main` push(러너 자동 배포).

## 6. 범위에서 제외

- 파일별 취소(한 파일만 빼고 계속) — 백엔드 finalize/삭제 변경이 필요하므로 이번엔 제외.
- 백엔드/`api.ts` 변경, 2단계 P2P.
- 업로드 재개(브라우저 새로고침 후 이어받기) 같은 세션 영속 — tus 오프셋 재개는 같은 세션 내에서만.
